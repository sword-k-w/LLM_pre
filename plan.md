## 演讲者A：提示工程——需求规格说明的工程学

> **总时长：10 分钟。** 本节演讲词可直接照讲，`【】` 内为舞台提示，不需要说出来。

---

### 开场（约 1 分钟）

"我先问大家一个问题。如果你要让程序员帮你做一个功能，你说——'做一个好用的登录页'——和——'做一个登录页，包含邮箱输入框和密码输入框，失败三次锁定账号，成功后跳转 /dashboard'——哪个说法更容易拿到你想要的结果？

LLM 和程序员面临的是完全同样的问题。它收到你的输入，要在一个巨大的可能性空间里推断你真正想要什么。你给的信息越模糊，它猜的方向就越多，输出质量的方差就越大。

Prompt Engineering，本质上不是学'说话技巧'——它是**需求规格说明（specification）**的工程实践。"

---

### 核心洞察（约 1.5 分钟）

"传统软件工程里，需求规格说明要弥合一个 gap：'你以为你想要的'和'工程师理解到的'之间的 gap。Prompt Engineering 解决的是同一个 gap，只是对象从工程师变成了语言模型。

这意味着好的 prompt 不是越长越好，也不是加魔法词，而是把三件事说清楚：**你想要什么（意图）、用什么方式推导（推理）、答案长什么样（格式）**。这就是我接下来要讲的三个规格维度。"

---

### 维度一：意图规格（约 2 分钟）

"第一个维度：意图规格——你要做什么，不要做什么，边界在哪。

模型的默认行为是'尽力帮你'，但它对'尽力帮你'的理解跟你的理解可能完全不一致。比如你说'帮我总结这篇文章'——它可能给你 100 字，也可能给你 500 字；可能给你要点，也可能给你段落摘要。没有一种对，但只有一种是你想要的。

意图规格要做三件事：

**角色定义**：告诉模型它是谁，限制它的语气、视角和知识边界。比如'你是课程助教，面向本科生回答，不要假设对方有研究背景'——这一句能大幅收窄模型的默认风格。

**任务边界**：告诉它要做什么，更重要的是**不要做什么**。'只回答课程相关问题，其他问题一律说超出范围'——没有这个约束，模型会热情地回答一切，包括它不该回答的。

**拒答规则**：没有把握时说不确定，比让它硬答然后产生幻觉要好得多。'资料没有覆盖时，明确说不确定，不要编造答案'——这一句直接影响系统的可信度。

把这三件事写进 system prompt，模型的行为就从'概率最高的通用回答'变成了'在你设定的约束内的最优回答'。"

---

### 维度二：推理规格（约 2 分钟）

"第二个维度：推理规格——让模型把推理过程显式化。

LLM 有一个特点：它在生成每个词的时候，只能看到已经生成的内容。如果你让它直接输出答案，它实际上是在'跳步'——没有草稿纸，直接写答案。对于简单问题没问题，但对于需要多步推导的任务，这会导致中间推理出错，而最终答案听起来却很有信心。

这就是思维链（Chain of Thought）要解决的问题。原理很简单：通过要求模型'先列步骤再给结论'，你等于给了它一张草稿纸。它必须把中间推理写出来，每一步推理都可以被后续步骤检验。

现代强模型已经在内部做了一定程度的思维链，但对于复杂的分析任务，显式要求它分步推理仍然能显著提高准确率。

实际怎么写：在提示里加'请先列出你的推理步骤，再给出结论'；或者针对特定任务说'先逐条检查每个前提是否成立，再判断论证是否有效'。关键是让推理过程可见，而不是藏在答案背后。"

---

### 维度三：输出规格 + 互动环节（约 3 分钟）

"第三个维度：输出规格——答案长什么样。

这是最直接影响工程可用性的维度。如果 AI 的回答要被代码解析、要填进表格、要被人快速扫读——你需要精确控制输出结构。

**格式约束**：直接描述你想要的结构。比如'只返回有效 JSON，不要 markdown 代码块，字段包括 name、score、reason'——这一句能省去大量后处理代码。

**少样本示例**：当格式描述不够直观时，给一两个输入→输出的例子。模型对模式的模仿能力非常强，一个好例子顶得上三句描述。少样本本质上不是一个独立的技巧，而是在格式用语言说不清楚的时候，用例子来补充规格。

【互动环节，约 1.5 分钟】

好，我们来做一个现场测试。

【展示一个糟糕的 AI 回答，内容是对'课程项目要交什么'的泛泛回应，比如'课程项目通常需要提交课件和报告，具体看老师要求'】

先问大家：这个回答哪里不好？

【等听众回答，预期答案：太模糊、没有具体信息、没有说不确定、格式随意】

那如果你来改提示词，加一句什么能让它好一点？

【等听众提议，引导向：加角色/加格式要求/加拒答规则，可以请 1-2 位同学说】

好，我们把这些加进去之后，来看看效果的变化。

【切到 demo 页，运行 baseline → prompt_optimized 对比，指出结构变整齐了、加了不确定声明】"

---

### 过渡到演讲者B（约 0.5 分钟）

"通过刚才的对比可以看到：加了意图、推理和格式规格之后，模型的回答结构变好了，边界也清晰了。

但你们注意到没有——它对课程的具体要求仍然是模糊的、或者说不确定的。这不是 prompt 写得不够好，而是 prompt 本身解决不了这个问题。模型只能表达它训练时学到的东西，它不知道我们这门课的评分标准，不知道老师上周的通知，不知道这份课程大纲的具体版本。

这个问题——如何把外部知识送进模型——就是下一位演讲者要讲的 Context Engineering 的核心。"

## 演讲者B：上下文工程——知识的架构

谢谢。如果提示工程是教模型*如何思考*，那么**上下文工程**就是给模型*提供要了解的知识*。

### 1. 核心动机：开卷考试
想象一下参加高级医学考试。无论你多么聪明，如果你完全依赖记忆（LLM的参数化权重），你可能会记错剂量。但是如果你桌上有一本开放的医学教科书（上下文），你会100%准确。上下文工程是一门将正确的"教科书页面"在模型需要的时刻放到它桌上的科学。

### 2. 我们面临的问题与解决方案
*   **问题1：幻觉和过时数据。**当LLM不知道答案时，它会自信地编造事实。
    *   *解决方案：***RAG（检索增强生成）。**不是直接询问模型，而是拦截用户的问题，搜索私有数据库（如公司的PDF手册）以获取答案，并将该文本注入提示中。然后我们说：*"仅根据提供的文本，回答用户。"*
*   **问题2：上下文窗口限制和令牌成本。**你无法将10,000页的数据库粘贴到提示中。即使可以，API成本将是天文数字，处理它也需要几分钟。
    *   *解决方案：***向量嵌入和分块。**我们将文档切分成小段落，将它们转换为数学向量，只检索与用户问题数学上最接近的前3个段落。
*   **问题3："迷失在中间"现象。**研究表明，如果你给AI 50页文本，它会在开头和结尾高度关注，但完全忽略埋在中间的信息。
    *   *解决方案：***重排序。**在检索上下文后，我们使用次要的轻量级算法对上下文进行排序，确保最相关的句子被放置在提示的最顶部或最底部，然后再发送给LLM。

> *过渡："所以，我们有了很好的提示，并注入了完美的上下文。但我们如何知道它在10,000个不同用户中确实能持续工作呢？为此，演讲者C将介绍评估工程。"*

## 演讲者C：Harness Engineering——处理 Reliability 问题

### 1. 来源
大家好，接下来我将介绍软件开发中一个与人工智能交互的新概念：**Harness Engineering**。

这个概念最近在技术圈引起了轰动，其源头可以追溯到 OpenAI 在2026年2月11日发表的一篇博客文章：*Harness engineering: leveraging Codex in an agent-first world*。这篇文章描述了一个实验：一个小型工程团队构建了一个包含上百万行代码的复杂应用程序，而人类没有手写任何一行代码，这里的代码不仅是源代码，还包括所有的限制规则，模块职责描述，工具使用指南等。整个 git 仓库完全由 Prompts 驱动 LLM 生成。

因为这仍是一个新兴的范式，社区仍在争论什么是最佳实践，甚至“标准答案”也尚未完全定型。因此，今天我主要结合 Open AI 与其他几篇博客，分享我个人的观点，并试图提炼出部分 high-level 的 idea.

### 2. 动机与理念

**Harness engineering**试图解决的主要问题是**Reliability**，即 LLM 生成代码中的错误或不可行，不够好的逻辑。举个例子：在上学期 CPU 的设计过程中，如果我让它写一个简单的测试，它有可能调用一个不存在的 Assassyn/Chisel 测试函数，虽然意图正确——时钟前进一个周期，但接口是错误的。无论多么先进 LLM 都是一台老虎机，因此在代码生成中，Bug和幻觉是一种不可避免的数学必然。

实际上，这个问题在几年前就被注意到了。回想经典的 **Test-Driven Design pattern**：人类负责确定接口并预先写好严格的单元测试，然后让AI生成代码以通过这些测试。随着AI变得越来越快，生成代码的速度远超人类编写测试的速度，这种模式演变成了**Agent-Agent 测试**——也就是一个AI代理负责写代码，另一个AI代理负责写测试，而人类仅仅扮演 Supervisor 的角色。但无论哪种测试方法，其核心哲学始终一致：**利用确定性的工程工具——比如编译器、自动化测试框架、代码检查工具——来限制、校验、纠正 AI 概率性的输出。**

**Harness Engineering**更进一步，其概念从测试延拓到整个开发过程中除 Agent 以外的所有内容，包括脚手架搭建，Context 工程按需获取信息，人类工程师的参与情况设计等。在我看来，其主要强调两个概念：Constraints 和 Feedback Loop。

### 3. 具体解决方案

#### Constraints
  
Open AI 原文中有一句话："人类的注意力是最宝贵的资源。"将其花费在具体的单元测试上性价比过低，完全放手又和对可靠性的要求相悖，因此工程师要关注的是实现中重要的部分：Agent 的职责，权限与互动方式，各模块接口，名称空间与类型系统。简言之，boundaries / invariance in the chaos. 它们就是人类为 Agent 设置的约束，只要最终生成的结果满足约束，那么内部实现无关紧要。出现问题的方式是施加新的约束，而不是"Fine, I will do it myself."

实际工程中有以下几个推荐策略：

第一，**接口优先设计（Interface-First Design）**。人类工程师的职责转变为架构定义。我们编写空洞的API端点、严谨的数据模式（Schemas）和严格的类型定义（Type Definitions）。我们精确规定什么数据可以输入，什么数据必须输出。AI代理的作用，仅仅是在这些极其坚固的墙壁之间填补逻辑。

第二，**互动与权限**。AI代理容易寻找阻力最小的路径，举个例子，如果让一个 Agent 负责编写测试并编写代码，那么很有可能测试不通过后它不会处理代码而是修改测试，同时它很有可能使用极简单的测试蒙混过关。方法关键在于将职责分配给两者且要求其互相不能访问对方编辑的文件，模拟真实情况下开发与测试的行为。

第三，**上下文的渐进式披露（Progressive Context Disclosure）**。早期的开发者常犯的一个错误是把整个百万行的代码库一次性塞给AI。尽管现在的上下文窗口已经足够大，但过多的噪音会严重干扰AI的推理能力。优秀的脚手架就像一个智能过滤器，它每次只向AI提供当前微小任务所绝对需要的接口、文件摘要和类型定义。少即是多，克制才是精准的前提。

#### Feedback Loop

然而，仅有约束不够，AI不能保证一次写出接口的正确实现。这就引出了第二个概念：**Feedback Loop**。

在整个项目的建构过程中，AI进行自我纠错的过程是必须的。举个可能的实现例子：

Agent A 生成一段代码后，harness 自动将这段代码放入一个完全隔离的、安全的沙盒环境，如临时容器（可能通过 github workflow 或是再并发一个 Agent 实现）。接着，系统会自动编译它，运行测试用例。如果它失败了——实际上它经常失败——这个闭环系统会自动捕获完整的堆栈跟踪信息、编译器报错或是日志。然后，系统会将这些绝对确定性的错误信息作为下一次的输入直接反馈给AI代理，并附上类似的提示：“你之前的实现由于这个具体的错误而崩溃，请修复你的逻辑。”

这就创造了一个完全自主的迭代引擎：生成 -> 编译 -> 失败 -> 错误反馈 -> 重新生成。在这个过程中，人类根本不需要去审查那段损坏的代码，系统自己完成了纠错的编排。

但是，在这个循环中存在一个致命危险：**反复迭代**。有时候，AI代理为了修复一个Bug尝试了五次，但它陷入了某种固执的逻辑盲区，每一次尝试都让代码变得比上一次更加糟糕、更加混乱。一个成熟的反馈循环系统必须设置停止阈值。当检测到代理连续失败，harness 应当中止当前的尝试，清空被污染的上下文缓存，将代码库回滚到上一个已知的完美状态，最终强制AI换一种完全不同的思路重新开始或引入人类参与。

### 4. Conclusion

虽然这个理念仍在不断成熟和发展中，但它的最终目标已经无比清晰。我们正在搭建一座桥梁，连接两个截然不同的世界：一边是 LLM 高效但充满概率的不确定世界；另一边则是软件开发对确定性的追求。初级工程师已经不复存在。对于今天的开发者来说，真正的问题已经不再是“我该如何编写这段代码？”而是：

**“我该如何构建一个好用的 harness，以确保AI能准确无误地替我写出这段代码？”**

> *过渡："光说不练假把式。" 在我们理解了 prompts engineering, context engineering 与 harness engineering 之后，演讲者D将向你展示它们在现实中如何工作。"*

## 演讲者D：现场演示——构建稳健的AI助手

谢谢。在接下来的15分钟里，我们将为一家虚构的电子商务公司"TechStore"构建客户支持AI。我们将实时进行（使用Python脚本/Jupyter Notebook/或像Cursor这样的IDE）。

### 第一阶段：原始模型（基准失败）
*   *操作：*我将向API发送一个简单的用户查询：*"我昨天购买的iPhone的退货政策是什么？"*
*   *结果：*AI回答了一个通用的30天退货政策。
*   *问题：*TechStore实际上对电子产品有严格的14天退货政策。AI**基于一般互联网数据产生了幻觉**。

### 第二阶段：应用提示工程
*   *操作：*我将更新系统提示。
    ```text
    角色：你是TechStore一个严格但有礼貌的客户服务代理。
    约束1：永远不要猜测政策。如果你不知道，说"我必须转接到人工服务。"
    输出：准确地用2个句子回答。
    ```
*   *结果：*AI现在说："很抱歉，但我无法访问TechStore特定的电子产品政策。我将转接到人工服务。"
*   *成果：*我们阻止了幻觉，但机器人仍然没有帮助。它缺乏知识。

### 第三阶段：应用上下文工程（RAG）
*   *操作：*我们模拟向量数据库检索。用户询问iPhone退货。我们的代码搜索我们的手册并提取一个文本块：*[TechStore政策数据：所有智能手机有14天的退货窗口。必须包含原始收据。]*
*   *操作：*我们将此动态地注入提示中。
*   *结果：*AI回应："对于您的iPhone，TechStore提供14天的退货窗口。请确保您随身携带原始收据。"
*   *成果：*模型现在是完全事实性、礼貌和严格格式化的。

### 第四阶段：应用评估工程
*   *操作：*我将运行一个脚本，针对5个不同的用户查询测试此管道（例如，询问鞋子、问一个有毒的问题、用法语询问）。
*   *结果：*出现一个终端输出，显示：`测试通过：5/5。平均延迟：1.2秒。幻觉率：0%。`

### 结论
