Problem Shift

真实系统的失败点，往往不在一句提示词里

只会“改提示词”不够

  • 回答看似流畅，但不引用课程资料
  • 同一问题多次提问，输出风格和质量漂移
  • 换一个任务后，之前的调优经验很难复用

工程视角要处理三层问题

  • Prompt：怎么说，模型更容易按要求答
  • Context：给什么信息，模型才有依据
  • Harness：怎么比较、评测、复现实验

Section 1

Prompt Engineering

目标不是”写花哨提示词”，而是精确描述任务（Specification）——缩小模型的输出空间，使它只能朝你要的方向走。

同一个任务，两种表述

  • 模糊：”做一个好用的登录页”
  • 精确：”做登录页：含邮箱框和密码框，失败三次锁定，成功后跳转 /dashboard”

给的信息越模糊，模型猜的方向就越多，输出质量的方差就越大。

黄金法则

把提示词给刚入职的同事看，让他按字面意思执行——如果他会懵，模型也会懵。

Three Dimensions

一个好的 Prompt 要把三件事说清楚

任务说明

你要做什么、不要做什么、边界在哪。

推理引导

让模型把推理过程显式化，而不是直接跳答案。

格式控制

答案长什么样：格式、结构、字段约束。

好的 Prompt 不是越长越好，也不是加魔法词——而是把这三件事写清楚。

Dimension 1 — Task

任务说明：告诉模型做什么、不做什么、边界在哪

角色定义

限制语气、视角和知识边界。

✗ "你是一个助手，回答用户的问题"

✓ "你是课程助教，面向本科生，不假设研究背景，用类比帮助理解"

任务边界

要做什么，更重要的是不要做什么——没有约束，模型会热情地回答一切。

✗ "帮用户解答所有问题"

✓ "只回答课程相关问题，其他问题一律说超出范围"

拒答规则

没有把握时说不确定，比让模型硬答再产生幻觉要好得多。

✗ （不写拒答规则，让模型自由发挥）

✓ "资料没有覆盖时，明确说不确定，不要猜测或编造"

Dimension 2 — Reasoning

推理引导：给模型一张草稿纸

LLM 生成每个词时只能看到已生成的内容。不给它"打草稿"的空间，它会跳步——答案听起来很有信心，但中间推理可能出错。

思维链（Chain of Thought）

要求模型"先列步骤再给结论"——强制把中间推理写出来，每一步都可以被后续步骤检验。

对比：加不加推理引导

✗ "小张比小李大 2 岁，小李比小王小 3 岁，小王 25 岁，小张多大？"

✓ "请先逐步写出每人的年龄关系，再给出答案。（同上题）"

加了引导后模型必须写出：小王 25 → 小李 22 → 小张 24，每步可验。

Dimension 3 — Output

格式控制：直接影响工程可用性

正向描述格式

说你想要什么，而不是不想要什么。

✗ "不要用 markdown"

✓ "用流畅的段落回答，不使用列表或标题"

✗ "返回 JSON"

✓ "只返回有效 JSON，字段：name、score（0–100 整数）、reason（一句话）"

少样本示例（Few-shot）

当格式用语言说不清时，给一两个输入→输出的例子。

输入："苹果公司 Q2 营收 940 亿，同比增长 5%"

输出：{"name":"苹果公司","score":72,"reason":"营收稳健但增速放缓"}

一个好例子顶得上三句描述。

Transition

Prompt 写好了，但还不够

即使把任务说明、推理引导和格式控制都写清楚，模型仍然不知道——我们课程的评分标准、老师的通知、这份课程大纲的具体版本。

Prompt 解决不了的问题

  • 模型只能表达训练时学到的知识
  • 它不知道你的私有资料和实时数据
  • 更好的 Prompt 只能让它"更好地不知道"

把外部知识送进模型，是下一个问题——Context Engineering。

Section 2

Context Engineering

Prompt 解决“怎么说”，Context 解决“给什么信息、按什么顺序给”。

在课程问答里，常见上下文来源

  • 系统提示：课程助手的身份和边界
  • 课程资料：讲义、作业说明、评分标准
  • 对话历史：之前追问了什么
  • 工具输出：检索结果、数据库字段、网页片段

Context Design

上下文工程不是“塞更多字”，而是“塞对信息”

关键问题

  • 哪些信息该给，哪些信息会造成噪声
  • 信息以什么顺序进入模型
  • 不同来源冲突时，谁优先
  • 长上下文里，哪些内容值得保留

课程问答中的具体做法

  • 优先塞与当前问题直接相关的课程片段
  • 保留出处，让回答能引用证据
  • 资料缺失时明确说“不确定”而不是编造

Section 3

Harness Engineering

当 prompt 和 context 开始变多，就需要一个“实验框架”来比较方案、固定样例、追踪退化。

它做的不是“让模型更聪明”

  • 让不同方案可复现、可横向比较
  • 把关键任务固化成标准测试样例
  • 把系统从一次性试玩变成能迭代的产品

Minimal Harness

轻量落地版：一套固定问题 + 多种模式 + 清晰检查项

固定问题集

确保每次实验都在回答同一批关键问题。

模式切换

一键比较 baseline / prompt / context / harness。

最小评测标准

是否答到点、是否引用资料、是否承认不确定。

轻量工作流

需要时再加入检索、工具调用、结果后处理。

Synthesis

一张图看清 Prompt / Context / Harness 的边界

Prompt

定义单次交互的任务、角色、格式和约束。

Context

组织进入模型的信息流，包括资料、状态、检索结果和对话历史。

Harness

为系统提供实验面板、回归样例、模式对比和最小评测。

关系可以理解为：Prompt 决定表达，Context 决定依据，Harness 决定迭代方式。

Checklist

希望同学带走的 6 条实践原则

  • 先写清输出结构，再追求文风和润色
  • 不给依据，就不要期待稳定高质量回答
  • 上下文越多不一定越好，相关性更重要
  • 把拒答规则写进去，比让模型硬答更可靠
  • 重要任务要有固定样例，避免优化后偷偷退化
  • 先做轻量 harness，再考虑更复杂的 agent 编排
