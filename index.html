真实系统的失败点，往往不在一句提示词里

只会“改提示词”不够

Section 1

Prompt Engineering

目标不是”写花哨提示词”，而是精确描述任务（Specification）——缩小模型的输出空间，使它只能朝你要的方向走。

同一个任务，两种表述

  • 模糊：”做一个好用的登录页”
  • 精确：”做登录页：含邮箱框和密码框，失败三次锁定，成功后跳转 /dashboard”

给的信息越模糊，模型猜的方向就越多，输出质量的方差就越大。

黄金法则

把提示词给刚入职的同事看，让他按字面意思执行——如果他会懵，模型也会懵。

Four Dimensions

Prompt 设计的四个核心维度

任务说明

角色、边界与拒答规则——定义模型"能做什么、不能做什么"。

推理引导

让模型把思考过程显式化，避免在多步推导上直接跳步出错。

格式控制

正向描述输出结构——字段、长度、是否使用 Markdown。

示例设计

Zero-shot 到 CoT Few-shot——用示例展示期望行为，而不只是描述。

Dimension 1 — Task

任务说明：告诉模型做什么、不做什么、边界在哪

角色定义

限制语气、视角和知识边界。

✗ "你是一个助手，回答用户的问题"

✓ "你是课程助教，面向本科生，不假设研究背景，用类比帮助理解"

任务边界

要做什么，更重要的是不要做什么——没有约束，模型会热情地回答一切。

✗ "帮用户解答所有问题"

✓ "只回答课程相关问题，其他问题一律说超出范围"

拒答规则

没有把握时说不确定，比让模型硬答再产生幻觉要好得多。

✗ （不写拒答规则，让模型自由发挥）

✓ "资料没有覆盖时，明确说不确定，不要猜测或编造"

Dimension 2 — Reasoning

推理引导：给模型一张草稿纸

LLM 生成每个词时只能看到已生成的内容。不给它"打草稿"的空间，它会跳步——答案听起来很有信心，但中间推理可能出错。

思维链（Chain of Thought）

要求模型"先列步骤再给结论"——强制把中间推理写出来，每一步都可以被后续步骤检验。

对比：加不加推理引导

✗ "小张比小李大 2 岁，小李比小王小 3 岁，小王 25 岁，小张多大？"

✓ "请先逐步写出每人的年龄关系，再给出答案。（同上题）"

加了引导后模型必须写出：小王 25 → 小李 22 → 小张 24，每步可验。

Dimension 3 — Output

格式控制：直接影响工程可用性

正向描述格式

说你想要什么，而不是不想要什么。

✗ "不要用 markdown"

✓ "用流畅的段落回答，不使用列表或标题"

✗ "返回 JSON"

✓ "只返回有效 JSON，字段：name、score（0–100 整数）、reason（一句话）"

少样本示例（Few-shot）

当格式用语言说不清时，给一两个输入→输出的例子。

输入："苹果公司 Q2 营收 940 亿，同比增长 5%"

输出：{"name":"苹果公司","score":72,"reason":"营收稳健但增速放缓"}

一个好例子顶得上三句描述。

Dimension 4 — Demonstration

示例设计：用展示代替描述

Zero-shot

只给指令，不给例子。任务边界清晰时优先选它——token 成本最低。

One-shot

一个完整的输入→输出例子，帮模型锚定语气、视角和细粒度格式要求。

Few-shot

2–5 个例子，形成完整模式。适合格式复杂、风格微妙或评判标准难以用语言表达的任务。

CoT Few-shot

例子本身包含推理过程，引导模型在新问题上也显式推理，比单纯说"请一步步思考"更可靠。

代价：例子越多，上下文越长，token 成本越高。一个质量差的例子比没有例子更糟——会把模型锚定到错误方向。

Transition

Prompt 写好了，但还不够

即使把任务说明、推理引导和格式控制都写清楚，模型仍然不知道——我们课程的评分标准、老师的通知、这份课程大纲的具体版本。

Prompt 解决不了的问题

  • 模型只能表达训练时学到的知识
  • 它不知道你的私有资料和实时数据
  • 更好的 Prompt 只能让它"更好地不知道"

把外部知识送进模型，是下一个问题——Context Engineering。

Part 02

上下文工程：
知识的架构

如果提示工程是教模型“如何思考”，那么上下文工程就是给模型“提供要了解的知识”。

  • 动机：架起静态模型权重与动态现实世界事实之间的桥梁
  • 要解决：幻觉 / 上下文窗口限制 / “迷失在中间”
  • 关键词：RAG、分块与向量嵌入、重排序与放置策略
Presenter B
核心比喻 开卷考试
一句话 把“正确页”放桌上

Mental Model

开卷考试：你不是更聪明，而是更“会找”

想象参加高级医学考试：如果完全依赖记忆（模型参数化权重），可能会记错剂量；但桌上放着教科书（上下文），就能更稳定、更准确。

上下文工程的目标：在模型需要的时刻，把正确的“教科书页面”放到它桌上。

现场互动（30秒）

给你一本 1000 页教科书，限时 30 秒回答一个具体问题——你会怎么做？

  • 先确定关键词/同义词（提取 query）
  • 查目录/索引/书签（检索）
  • 快速扫读定位段落（分块 + 相似度）
  • 把最相关页摊在桌上（放置与排序）

对应到 LLM 系统

  • 检索：从文档库中找最相关片段
  • 压缩：只带上有限的关键证据
  • 编排：让证据出现在模型“看得到”的位置

Problem 01

幻觉与过时数据：模型不知道，就会“自信地编”

当 LLM 缺少事实依据时，它倾向于给出听起来合理的答案，但这些“事实”可能是编造的或已经过时。

解决方案：RAG（检索增强生成）

  • 拦截用户问题（query）
  • 检索私有资料库（PDF、手册、课程资料、知识库）
  • 把检索到的文本作为上下文注入提示
  • 给出硬约束：“仅根据提供的文本回答”

最小管道（概念图）

  • User：问题
  • Retriever：找证据（Top-K 段落）
  • Prompt Assembler：把证据放进上下文
  • LLM：基于证据生成答案

Example

同一个问题：
没有上下文 vs 注入上下文

我们不指望“模型自己知道”，而是让它在回答前先“翻到正确页”。

没有上下文（易幻觉）

Q：我昨天购买的 iPhone 的退货政策是什么？

A：一般情况下电子产品支持 30 天无理由退货……（可能不符合公司实际政策）

注入“政策片段”（可追溯）

[TechStore 政策数据]
所有智能手机有 14 天的退货窗口。
必须包含原始收据。

指令：仅根据提供的政策数据回答。

A：TechStore 对智能手机提供 14 天退货窗口，请确保保留并提供原始收据。

Problem 02

上下文窗口限制与令牌成本：
不可能把 10,000 页都塞进去

把所有资料一次性粘贴到提示里不仅做不到，也不经济：窗口有限、成本高、延迟长，还会降低相关性。

解决方案：分块 + 向量嵌入

  • 将文档切分为小段落（chunks）
  • 把每个 chunk 转换为向量（embeddings）
  • 检索与问题在向量空间最接近的 Top-K
  • 只携带 Top-K 作为上下文（更省、更快、更相关）

设计要点

  • 分块：太大→浪费窗口；太小→语义断裂
  • Top-K：太多→稀释注意力；太少→证据不足
  • 去重/合并：相近片段合并，减少噪声

Problem 03

“迷失在中间”：
找到了证据，不等于模型看见了

Stanford Liu et al. (2023) 实验：把正确答案藏在不同位置，正确率呈 U 形分布——开头和结尾最高，中间最低。

类比：会议发言里，第一句和最后一句最容易被记住；中间第 14 条论点往往就没人记得了。

解决方案：重排序 + 关键事实前置

  • 检索后按相关性重排序，最重要的 chunk 放首位或末位
  • 对关键原文单独提炼成“关键事实：……”一行，放在上下文最前面
  • 避免把最有价值的证据埋在中间长段落里

一句话总结

“不只是找到什么，更要决定怎么摆——信息的位置决定模型是否真正'看见'它。”

Checklist

上下文工程的交付物：
让“信息流”可控、可复现

你要设计的不是一段文本，而是一条把信息送进模型的流水线。

  • 定义知识源：课程资料 / 手册 / 数据库 / 工具输出
  • 分块与索引：chunk 策略 + embeddings + 向量库
  • 检索策略：Top-K、过滤条件、去噪与去重
  • 编排策略：重排序、放置（头/尾）、必要的压缩摘要
  • 回答约束：仅基于证据回答；缺证据就明确说“不确定/缺口”
  • 可观测性：记录“用了哪些上下文块”，便于复现与调试

Section 3

Harness Engineering

当 prompt 和 context 开始变多，就需要一个“实验框架”来比较方案、固定样例、追踪退化。

它做的不是“让模型更聪明”

  • 让不同方案可复现、可横向比较
  • 把关键任务固化成标准测试样例
  • 把系统从一次性试玩变成能迭代的产品

Minimal Harness

轻量落地版：一套固定问题 + 多种模式 + 清晰检查项

固定问题集

确保每次实验都在回答同一批关键问题。

模式切换

一键比较 baseline / prompt / context / harness。

最小评测标准

是否答到点、是否引用资料、是否承认不确定。

轻量工作流

需要时再加入检索、工具调用、结果后处理。

Synthesis

一张图看清 Prompt / Context / Harness 的边界

Prompt

定义单次交互的任务、角色、格式和约束。

Context

组织进入模型的信息流，包括资料、状态、检索结果和对话历史。

Harness

为系统提供实验面板、回归样例、模式对比和最小评测。

关系可以理解为：Prompt 决定表达，Context 决定依据，Harness 决定迭代方式。

Checklist

希望同学带走的 6 条实践原则

  • 先写清输出结构，再追求文风和润色
  • 不给依据，就不要期待稳定高质量回答
  • 上下文越多不一定越好，相关性更重要
  • 把拒答规则写进去，比让模型硬答更可靠
  • 重要任务要有固定样例，避免优化后偷偷退化
  • 先做轻量 harness，再考虑更复杂的 agent 编排
