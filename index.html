Problem Shift

真实系统的失败点，往往不在一句提示词里

只会“改提示词”不够

  • 回答看似流畅，但不引用课程资料
  • 同一问题多次提问，输出风格和质量漂移
  • 换一个任务后，之前的调优经验很难复用

工程视角要处理三层问题

  • Prompt：怎么说，模型更容易按要求答
  • Context：给什么信息，模型才有依据
  • Harness：怎么比较、评测、复现实验

Section 1

Prompt Engineering

目标不是“写花哨提示词”，而是把任务定义、约束条件和输出规格说清楚。

最常用的四个杠杆

  • 角色：你希望模型扮演谁
  • 任务：要完成什么、不要做什么
  • 格式：答案结构、长度、字段约束
  • 示例：给少量样例，减少理解偏差

Prompt Template

一个好 prompt 至少应该回答四件事

1

你是谁

例如：你是课程助教，回答应面向本科生。

2

你要做什么

例如：只回答与课程资料相关的问题。

3

你如何输出

例如：先给结论，再列证据来源。

4

你何时拒答

例如：资料没有覆盖时，明确说明不确定。

Section 2

Context Engineering

Prompt 解决“怎么说”，Context 解决“给什么信息、按什么顺序给”。

在课程问答里，常见上下文来源

  • 系统提示：课程助手的身份和边界
  • 课程资料：讲义、作业说明、评分标准
  • 对话历史：之前追问了什么
  • 工具输出：检索结果、数据库字段、网页片段

Context Design

上下文工程不是“塞更多字”，而是“塞对信息”

关键问题

  • 哪些信息该给，哪些信息会造成噪声
  • 信息以什么顺序进入模型
  • 不同来源冲突时，谁优先
  • 长上下文里，哪些内容值得保留

课程问答中的具体做法

  • 优先塞与当前问题直接相关的课程片段
  • 保留出处，让回答能引用证据
  • 资料缺失时明确说“不确定”而不是编造

Section 3

Harness Engineering

当 prompt 和 context 开始变多，就需要一个“实验框架”来比较方案、固定样例、追踪退化。

它做的不是“让模型更聪明”

  • 让不同方案可复现、可横向比较
  • 把关键任务固化成标准测试样例
  • 把系统从一次性试玩变成能迭代的产品

Minimal Harness

轻量落地版：一套固定问题 + 多种模式 + 清晰检查项

固定问题集

确保每次实验都在回答同一批关键问题。

模式切换

一键比较 baseline / prompt / context / harness。

最小评测标准

是否答到点、是否引用资料、是否承认不确定。

轻量工作流

需要时再加入检索、工具调用、结果后处理。

Synthesis

一张图看清 Prompt / Context / Harness 的边界

Prompt

定义单次交互的任务、角色、格式和约束。

Context

组织进入模型的信息流，包括资料、状态、检索结果和对话历史。

Harness

为系统提供实验面板、回归样例、模式对比和最小评测。

关系可以理解为：Prompt 决定表达，Context 决定依据，Harness 决定迭代方式。

Checklist

希望同学带走的 6 条实践原则

  • 先写清输出结构，再追求文风和润色
  • 不给依据，就不要期待稳定高质量回答
  • 上下文越多不一定越好，相关性更重要
  • 把拒答规则写进去，比让模型硬答更可靠
  • 重要任务要有固定样例，避免优化后偷偷退化
  • 先做轻量 harness，再考虑更复杂的 agent 编排
