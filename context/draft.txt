第二部分：上下文工程 13 分钟演讲稿

大家好，前面一部分已经讲了 Prompt Engineering，也就是我们怎么把任务、角色、格式和约束讲清楚，让模型更容易按我们的要求输出。接下来我负责第二部分，Context Engineering，也就是上下文工程。

如果说 Prompt Engineering 解决的是“怎么说”，那么 Context Engineering 解决的就是“给模型什么信息，以及这些信息按什么顺序给进去”。

我想先给大家一个很直观的类比。大家可以想象一下，你现在要参加一场开卷考试。假设这门课内容非常多，教材有一千页，考试时限只有三十秒。这个时候，你真正需要的，不是把这一千页全背下来，而是能不能在有限时间内快速找到最相关的那几页，并把它们摊开放在眼前。上下文工程做的事情，其实就是这个。它不是让模型凭空变得更聪明，而是让模型在回答问题的时候，手边刚好有最需要的那几页资料。

所以，从工程视角看，上下文工程的核心动机，就是架起静态模型权重和动态现实世界事实之间的桥梁。大语言模型的参数里确实存储了大量知识，但这些知识有几个天然局限。第一，它可能过时。第二，它可能不包含你所在课程、公司、团队内部的私有信息。第三，即便它“好像知道”，也不一定能稳定、准确地调用出来。也就是说，模型本身并不能天然保证“知道最新事实”或者“知道你这门课的具体要求”。这时候，问题就不再是提示词写得够不够漂亮，而是你有没有把正确的信息送到模型面前。

接下来我想分三部分讲清楚上下文工程到底在解决什么问题。

第一类问题，是幻觉和过时信息。

大家应该都见过这种情况。你问模型一个它其实不太确定的问题，它不会老老实实说“我不知道”，而是很流畅地给你生成一个看上去挺合理、但实际上并不准确的答案。比如在我们这个课程场景里，如果你直接问：“小组授课最后要交什么？”如果模型没有拿到课程原始要求，它很可能会根据自己见过的大量教学项目经验，给出一个泛化的回答，比如“提交 PPT、口头汇报和书面报告”。这个回答听起来非常正常，但它不一定符合我们这门课的真实要求。

所以这里的关键点是，语言流畅不等于事实正确。模型说得越自然，有时反而越容易让人忽略它可能在编。

那怎么解决？最典型的方法就是 RAG，Retrieval-Augmented Generation，也就是检索增强生成。它的基本思路不是“直接拿问题问模型”，而是先拦截用户的问题，去搜索我们的知识库，比如课程资料、讲义、作业要求、FAQ、评分标准，找到和当前问题最相关的片段，然后把这些片段一起塞进 prompt 里，再告诉模型：请你只根据这些资料来回答。

这个过程非常重要，因为它把“回答问题”拆成了两个步骤。第一步，不是生成，而是找证据。第二步，才是在证据基础上生成回答。也就是说，模型不再主要依赖自己记忆里的模糊知识，而是优先依赖当前会话里被注入的证据。

我们这次 demo 里，其实也已经体现了这个思想。`demo.html` 里面可以勾选课程资料的上下文块，`server.py` 会把这些 context blocks 注入到消息里。于是同一个问题，在没有上下文的时候，模型可能只能给一个“像那么回事”的回答；一旦把课程资料片段送进去，回答就会变成“有依据的回答”。这就是 Context Engineering 最直观的价值。

第二类问题，是上下文窗口限制和令牌成本。

很多同学第一次接触上下文工程的时候，会有一个很自然的想法：既然给模型资料有用，那我把所有资料全塞进去不就行了吗？但真正做系统时很快会发现，这条路走不通。

原因很简单。第一，模型的上下文窗口是有限的。第二，输入越长，调用成本越高，延迟也越大。第三，就算一个模型支持很长上下文，也不意味着所有信息都值得塞进去。因为一旦信息太多、太杂，真正相关的内容反而会被稀释，模型的注意力会被无关信息拖走。

所以，上下文工程不是“往里塞更多字”，而是“只塞最相关的信息”。这背后的典型技术路线，就是分块和向量检索。

具体来说，我们不会把整本课程手册原封不动丢给模型，而是先把文档切成很多小块，也就是 chunks。每个 chunk 可以理解为一个语义相对完整的段落或者片段。然后，我们再把这些 chunks 转成向量表示。这样，当用户提出问题时，我们就可以把问题也转成向量，去向量空间里找和它最接近的前几个片段，也就是 Top-K 结果。最终送进模型的，不再是整本资料，而是最相关的三段、五段或者若干条证据。

这个设计背后体现的是一个很重要的工程思想：我们不是在优化“资料总量”，而是在优化“相关性密度”。

当然，这里面还有很多细节值得注意。比如 chunk 太大，会浪费窗口；chunk 太小，又可能把原本完整的语义切碎。再比如 Top-K 取太少，可能证据不够；取太多，又会引入噪声。所以真正做上下文工程时，我们调的往往不是一句提示词，而是一整套信息管道：怎么切，怎么检，检多少，怎么去重，怎么组织。

第三类问题，是“迷失在中间”。

这也是长上下文里一个特别典型、但又容易被忽视的问题。很多研究和实践都发现，当你给模型一大段很长的上下文时，它往往更关注开头和结尾，而中间部分的信息容易被忽略。也就是说，并不是你把关键证据放进去了，模型就一定会稳定使用它。信息的位置，也会影响模型是否真正“看见”它。

这就引出上下文工程里的另一个关键动作：重排序和编排。

假设我们已经通过检索拿到了五段相关文本，这时候不是简单按原顺序全塞进去就完了。我们还要进一步思考：哪一段最重要？哪一段最应该被模型先看到？哪些内容适合放在开头，哪些内容适合放在结尾？有没有必要先做一层轻量摘要，把最关键的事实压缩成一句话，放在一个更显眼的位置？

所以，上下文工程除了“找资料”，还有一层很重要的工作，叫信息编排。你可以把它理解成一种给模型“摆桌面”的能力。资料不是越多越好，而是摆放方式要合理。最重要的信息，要出现在最容易被注意到的位置。

讲到这里，我们可以把 Context Engineering 简单总结成一句话：它的目标不是增加信息量，而是提高证据的可用性。

下面我想把这个概念和我们这次的课程项目再对应一下，这样大家会更容易把理论和实际连接起来。

在 `index.html` 这套课件结构里，Prompt、Context、Harness 三个部分分工非常明确。Prompt 负责定义任务表达，Context 负责提供依据，Harness 负责比较和评估。到了我们第二部分，重点其实就在一句话上：没有依据的回答，再漂亮也不可靠。

比如在课程问答助手这个场景中，常见的上下文来源至少有四类。

第一类是系统级上下文，也就是模型的身份、边界、回答风格，比如“你是课程助教”“资料不足时要明确说明不确定”。第二类是课程资料本身，比如讲义、作业说明、评分标准、课程通知。第三类是对话历史，也就是前面对话已经问过什么、上下文是否发生了变化。第四类是工具输出，比如检索结果、数据库字段、网页抓取内容，甚至是别的程序给出的结构化结果。

而上下文工程做的事情，就是把这些来源进行筛选、排序和组合，决定哪些该进模型，哪些不该进，谁优先，谁作为补充，冲突时以谁为准。

这也是为什么我们说它是一门工程，而不是一个小技巧。因为它处理的已经不是一句 prompt，而是整个信息流。

如果大家后面要做自己的项目，我建议至少记住下面这几个实践原则。

第一，不要把上下文理解成“背景介绍”，而要把它理解成“证据”。凡是送进模型里的内容，都应该尽量和当前问题有直接关系。

第二，上下文不是越多越好，相关性优先于完整性。你宁可给三段高度相关的材料，也不要给三十段模糊相关的内容。

第三，要让模型知道证据边界。也就是说，如果资料没有覆盖，模型应该明确说“不确定”或者“资料不足”，而不是自己补全。

第四，要记录你到底给了模型哪些上下文。因为只有这样，回答出了问题时你才能复盘：到底是没检到、检错了，还是排布方式出了问题。

第五，把上下文工程和提示工程分开想。Prompt 决定模型怎么表达，Context 决定模型凭什么表达。两者互相配合，但不要混成一件事。

最后，我想用一句话作为我这一部分的收束。

Prompt Engineering 的本质，是约束模型的表达；Context Engineering 的本质，是组织模型可见的信息；而一个可靠的 LLM 系统，往往不是因为它“会说”，而是因为它“有依据地说”。

所以，当我们下一次觉得模型“怎么又答错了”的时候，不妨先别急着改提示词，先问自己一个问题：我到底有没有把它需要的那一页资料，真的放到它眼前？

谢谢大家。接下来我把内容交给下一位同学，他会继续讲 Harness Engineering，也就是当我们已经有了 prompt 和 context 之后，怎么去系统地比较、测试和评估这些方案是否真的稳定可靠。
