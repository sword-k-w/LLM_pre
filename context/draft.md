第二部分：上下文工程 13 分钟演讲稿

大家好，我来讲第二部分：上下文工程（Context Engineering）。

我先从一个失败案例开始。

假设你已经按上一部分讲的那样，认认真真写好了一个提示词——定义了角色、规范了格式、加了拒答规则。然后你把这个助手接到课程问答系统里，测了几个问题，效果不错，你很满意。

但有一天，有个同学问："这门课的小组授课到底要交什么？"

模型回答说："通常需要提交 PPT 文件、口头汇报录像，以及一份不少于2000字的书面报告。"

听起来很合理，对吗？——但这不是你们课程老师的真实要求。模型不知道这门课的具体规定，它给的是它从海量互联网文本里学到的"一般教学项目"的通用做法。语言流畅，结构完整，但对你毫无价值，甚至可能让你错过真正的要求。

这就是 Context Engineering 要解决的核心问题。

如果说 Prompt Engineering 解决的是"怎么说"——用什么角色、什么格式、什么约束来表达——那 Context Engineering 解决的就是"给什么信息，以及这些信息按什么顺序送进去"。Prompt 能约束表达方式，但不能凭空产生你课程文件里的具体内容。让模型有依据地说话，靠的是 Context。

在进入具体问题之前，我想先给大家建立一个思维模型。


**互动（30 秒）**

想象这样一个场景：你手上有一本1000页的教科书，内容很全；但现在限时30秒，要你回答一个具体问题——比如"第七章里关于某个定义的原文是什么"。

我想请大家快速想一下：这30秒里，你的第一个动作是什么？

（停顿 2 秒，环顾听众，等待 1-2 个回应）

大多数人会这样做：先认定几个关键词，翻目录找到大概章节，快速扫读定位段落，然后把那一两页摊开放在眼前——这才开始答题。

这整个流程，其实就是上下文工程在做的事。

提取关键词，对应的是"把用户问题转成检索查询"；翻目录定位，对应的是"在知识库里做向量检索"；把最相关的页面摊开，对应的是"把证据注入提示"。

上下文工程不是让模型凭空变聪明，而是让它在回答的时候，手边刚好有最需要的那几页资料。

好，我们就沿着这个类比，来看上下文工程要解决的三类问题。


**问题一：幻觉与过时数据**

刚才那个课程要求的例子，就是典型的幻觉。

大语言模型在不知道答案的时候，不会老实说"我不知道"，而是会根据训练数据里的统计规律，生成一个"看起来合理"的回答。这里有一个很重要的认知：语言流畅不等于事实正确。模型说得越自然，有时反而越容易让人忽略它可能在编。

解决这个问题最典型的技术路线叫做 RAG，Retrieval-Augmented Generation，也就是检索增强生成。它的核心思路是把"直接问模型"拆分成两个步骤——先找证据，再基于证据回答。

具体流程是这样的：

第一步，拦截用户问题，不直接送给模型，而是先拿着问题去搜索我们自己的知识库，比如课程讲义、作业说明、评分标准、FAQ 文档，找到最相关的若干段文本。

第二步，把这些文本片段作为上下文，和用户问题一起送进提示，同时给模型一个硬约束：只能根据提供的资料来回答，不允许自己补充或编造。

用流程图来说，这个最小 RAG 管道只有四个节点：

    用户问题 → 检索器（找出 Top-K 相关段落） → Prompt 拼装器（把段落塞进上下文） → LLM（基于证据生成答案）

这个设计最关键的地方不在于技术有多复杂，而在于它改变了模型的"信息来源"。模型不再优先依赖自己训练时的模糊记忆，而是优先读取当前会话里被注入的证据。就好比你考试时，不是靠背诵，而是先翻开教材，再根据教材作答。

这也是为什么我们这次 demo 里，`demo.html` 里可以勾选课程资料上下文块——这就是在模拟一个简化的 RAG 注入过程。没有上下文块时，模型回答靠的是自己的"模糊记忆"；加了上下文块之后，回答就有了可追溯的证据来源。


**问题二：上下文窗口限制与令牌成本**

理解了 RAG，很多同学可能会想：既然给模型资料有效果，那我把所有资料全塞进去不就行了？

这条路在工程上走不通，有三个实际约束。

第一，窗口有上限。就算是目前支持超长上下文的商用模型，窗口也是有限的，一份完整的课程手册或者一本技术文档，很可能直接超出限制。

第二，输入越长，成本越高、延迟越长。主流商用 API 按 token 计费，一次查询带上几千个 token 的背景资料和带上几十万个 token，成本可能相差几十到几百倍，响应时间也会明显拉长。

第三，信息太多反而降低质量。当上下文里堆满大量不直接相关的内容时，真正有用的证据会被噪声稀释，模型的注意力也会被拖走，回答准确性反而可能下降。

所以，上下文工程的核心原则不是"往里塞更多字"，而是"只塞最相关的信息"。

具体的技术路线叫做分块加向量检索。我们把文档切分成一个个语义相对完整的小段，每段通常几十到几百个 token，英文叫 chunk。然后用嵌入模型（embedding model）把每个 chunk 转成一个向量，存入向量数据库。当用户提问时，把问题也转成向量，在向量空间里计算相似度，找出数学上最接近的前 K 个 chunk，这 K 个 chunk 就是最终送进模型的上下文。

整套信息管道可以理解为：你不是在搬一整座图书馆，而是在找最相关的那几页。

这里有几个值得注意的调参细节：chunk 切太大会浪费窗口；切太小会把完整语义切碎，模型理解不了。Top-K 取太少证据可能不足；取太多又会引入噪声。实际工程里，这些参数都需要通过实验来找到合适的平衡。


**问题三："迷失在中间"**

假设我们现在已经通过检索拿到了五段相关文本，是不是直接拼起来送进模型就好了？

很遗憾，不是。这里有一个研究层面已经被明确验证过的问题，叫做"迷失在中间"（Lost in the Middle）。这来自斯坦福大学 Nelson Liu 等人 2023 年的一篇论文。

他们做了这样一个实验：给模型一段很长的上下文，把正确答案藏在不同位置——有时在开头，有时在结尾，有时在正中间。结果发现，正确答案放在开头或结尾时，模型的回答准确率明显更高；而放在中间位置时，准确率会显著下降。

用一个日常类比来理解这个现象：你在一场会议里发言，你说的第一句话和最后一句话最容易被人记住，但中间第14个论点往往就没人记得了。这不是因为那个论点不重要，而是人类对序列信息的处理天然存在这种注意力分布——模型也不例外。它对序列开头和结尾的"注意力"更高，中间部分容易被忽略。

这意味着：仅仅"找到了相关证据"是不够的，还要关心这段证据被放在了什么位置。

具体的解决方案有两个。

第一个叫重排序（Re-ranking）。检索到若干相关 chunk 之后，不是按原始顺序直接拼接，而是再做一次相关性评分，把最重要的 chunk 排到序列的最前面或最后面，而不是埋在中间。

第二个是关键事实前置。如果某段资料里有一句非常关键的原文，可以在提示开头专门提炼出来——比如加一行"关键事实：……"——而不是让它淹没在几百字的段落里。这样做的目的，是让模型在进入长文本之前，就已经"注意到"了最重要的信息。

总结一下：上下文工程不只是解决"找到什么"，还要解决"怎么摆放"。信息的位置，决定了模型是否真正"看见"它。


**和这次 demo 的对应**

让我把上面三个问题和我们的演示系统对应一下。

在 `demo.html` 里，你可以切换四种模式：baseline、prompt_optimized、context_augmented、harness_mode。从 baseline 到 context_augmented 这一步，本质上就是在演示今天讲的 RAG 思路——你勾选的上下文块，会被 `server.py` 注入到请求消息里，模型在收到问题时，这些资料已经被放在了它的"桌面上"。

同一个问题，在没有课程资料的时候，模型只能给出一个"听起来合理"的通用回答；注入课程资料之后，回答就变成了"有依据的回答"，可以引用具体内容，也可以在资料不足时明确说"缺口"。

这就是 Context Engineering 最直接的工程体现：不是改写提示词，而是设计信息流。


**几条带走的实践原则**

最后，我想给大家留几条做上下文工程时最有用的实践原则。

第一，把上下文理解成"证据"，而不是"背景介绍"。送进模型的每一段内容，都应该和当前问题有直接关系。

第二，相关性比完整性重要。三段高度相关的资料，比三十段模糊相关的内容好用得多。

第三，告诉模型证据边界。资料没有覆盖的地方，要在提示里明确要求模型说"不确定"，而不是让它自己补全。

第四，记录你给了模型哪些上下文。回答出问题的时候，你才能复盘：是没检索到、检索错了，还是排布顺序出了问题。

第五，把 Prompt 和 Context 分开设计。Prompt 决定怎么表达，Context 决定凭什么表达。两件事的优化方向不一样，不要混为一谈。


**结语**

最后一句话作为总结：一个可靠的语言模型应用，往往不是因为它"会说"，而是因为它"有依据地说"。Prompt Engineering 约束了模型的表达；Context Engineering 组织了模型可见的信息。两者配合，才构成稳定回答的基础。

所以，当你下一次发现模型回答不准确的时候，先别急着改提示词——先问自己一个问题：我有没有把模型需要的那一页资料，真的放到它眼前？
